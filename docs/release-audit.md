## Verdict

This repository is a substantial implementation baseline, but it is **not ready for a learner-facing Core Beta or a production NUC deployment**. The strongest implemented areas are the curriculum graph and authored-draft validators, deterministic learning/exam authority, encrypted credential vault, administrator plan revisions, executable data lifecycle, static GitHub reviewer, and local isolated-runner harnesses. The largest gaps are human review and approval of the authored curriculum, signed/deployed app and runner images plus KVM evidence, complete auth/session and learner-journey browser evidence, live deployment/restore evidence, live-provider AI evaluation, and accessibility/security/performance verification.

## Immediate launch blockers

1. **The authored corpus is not publication-ready.** There are 476 authored lessons, 476 assessment banks and 1,385 deterministic items (475 MCQ, 409 fill-gap, 500 code and 1 trace), including 500 visible and 574 hidden code cases. Every lesson and bank is still an AI-assisted draft, every reviewer is null, and 0 items are exam eligible. The publication gate correctly blocks release until independent human review and approval.
2. **The authoritative runner is locally execution-verified but not security-cleared or production-deployed.** Local immutable images and `/opt/runner/execute` harnesses now exist for C (Alpine GCC 14.2), C++ (Alpine G++ 14.2), Java (OpenJDK 21.0.11), Python (3.14.6) and JavaScript (Node 22.23.1). The exact five current images pass 17 runtime contracts, four real-executor checks and all 1,074 curriculum cases. Vulnerability scan, SBOM and release-record generation are **NOT RUN** for these new identities because the required local scanner is unavailable; the retained Trivy 0.69.2 zero-HIGH/CRITICAL reports bind older image digests and must not be treated as current clearance. Release still requires fresh exact-image security evidence, deployed KVM/NUC digests, adversarial isolation evidence, and reproduction of every corpus report on the deployed boundary.
3. **KVM isolation is documented but not demonstrated.** The code securely constructs Docker arguments, but `RUN-004`/`NFR-SEC-002` require a dedicated KVM VM and measured no-egress/no-secret/cross-job evidence. [`docs/runbooks/runner-isolation.md`](runbooks/runner-isolation.md) is an operator plan, not execution evidence.
4. **The invitation, password-reset, one-device and out-of-band lost-device boundaries now have real PostgreSQL evidence, but the complete authentication acceptance matrix is not closed.** Production auth fails closed; exact 24-hour bearer invitations are race/replay tested; request approval/rejection and password activation/reset have database journeys; one active browser family is database-enforced with token-free history, learner/admin controls and notices. A neutral public lost-device ceremony now keeps its 15-minute single-use mailbox proof hashed, derives it only in worker memory, carries it in a non-HTTP URL fragment, binds it to the owner/session, and still requires a separate identity check plus fresh-MFA/reasoned administrator approval. Google/linking, full TOTP/lost-factor recovery and the combined real-cookie sensitive-command matrix remain open. See the [auth recovery verification](evidence/auth-recovery-verification-2026-07-12.json), [`integration/auth-recovery.integration.test.ts`](../integration/auth-recovery.integration.test.ts), and [`src/lib/security/lost-device-recovery.ts`](../src/lib/security/lost-device-recovery.ts).
5. **Production email delivery is implemented but not proven.** [`scripts/process-outbox.ts`](../scripts/process-outbox.ts), [`Dockerfile`](../Dockerfile), and [`compose.yaml`](../compose.yaml) now wire the Gmail-capable retrying worker, but no configured Gmail OAuth delivery, SPF/DKIM/DMARC result, template receipt, bounce/suppression behavior, or failure-recovery artifact exists.
6. **A clean disposable backup/restore drill passes, but production and offsite recovery are not proven.** The synthetic Docker drill exercises encrypted backup, verification, direct isolated restore, automated restore reporting and cleanup without real data or durable keys. There is still no run against the actual 2 TB target, production-sized data, separately protected real key, Google Drive/rclone offsite copy, offsite download restore, signed quarterly operator report, or measured production RPO/RTO.
7. **A complete service-level PostgreSQL learner journey now passes, but the authenticated browser journey is not closed.** One explicitly human-reviewed synthetic fixture goes through the real review/evidence/publication gate, pointer-selected plan, diagnostic, lesson/session, wrong and assisted practice, confirmed remediation, E4/E5/E6 mastery gates, reviewed formal exam, private badge and delayed review. This proves production authority wiring without approving the committed draft corpus. Access/interrupted onboarding and the same placement → lesson → practice → remediation → exam → mastery → review path through learner browser UI remain open. See [`persisted-learner-journey-2026-07-12.json`](evidence/persisted-learner-journey-2026-07-12.json).
8. **Release verification remains incomplete.** There is no endpoint-wide auth/authorization E2E matrix, live-provider tutor-quality run, deployed all-provider outage/alert journey, real GitHub E2E, production runner fault/restart/KVM test, production retention execution report, ten-user/two-run load report, external security scan, or manual assistive-technology/device report. A deterministic zero-network PostgreSQL provider-outage journey now covers the local application behavior only.
