# Code-review remediation matrix — 2026-07-14

This document maps every High finding in the read-only [2026-07-13 codebase review](../LearnCoding-Code-Review.md) to the current local implementation and its closest behavioral test. It does not modify or reinterpret the original review.

## Status and evidence rules

- **18 of 20 High findings are fixed locally.** “Fixed locally” means the repository contains a concrete implementation change and focused automated coverage. It does not mean deployed, production-observed, or independently human-reviewed course content.
- **2 of 20 are partial release gates.** The static authorization inventory does not yet have a two-owner runtime test for all 87 authenticated operations, and the real-cookie browser harness does not yet cover OAuth, privileged administrator journeys, or every configured browser.
- A linked test is evidence that the behavior is specified and executable. Final command outcomes belong in the release audit; this matrix does not invent a passing run.
- No row claims production deployment, a physical-device test, live-provider quality, or human editorial approval of the course corpus.

## Medium findings: addressed versus deferred

The original review reports 70 Medium findings but only lists selected examples. This section therefore records the explicit examples that can be traced without pretending the unlisted sub-report exists in this repository.

### Addressed or materially mitigated

- Locale-dependent curriculum hashes were replaced with deterministic code-point ordering in [hash.ts](../src/lib/curriculum-publication/hash.ts), covered by [hash.test.ts](../src/lib/curriculum-publication/__tests__/hash.test.ts).
- Runner-client stdout/stderr size validation now enforces the submitted output budget in [client.ts](../src/lib/runner/client.ts), covered by [client.test.ts](../src/lib/runner/__tests__/client.test.ts).
- Assessment-correction audit and failure-recording errors now emit bounded operator events instead of disappearing in [worker.ts](../src/lib/assessment-corrections/worker.ts).
- The route fallback, tab semantics, dashboard observability, scheduler locking, and several idempotency/notification issues described as Medium are subsumed by the High-finding fixes above.

### Deferred or still requiring a separate release gate

- Production CSP still contains `script-src 'unsafe-inline'` in [next.config.ts](../next.config.ts); nonce/hash migration remains deferred.
- [db/client.ts](../src/lib/db/client.ts) still has a local fallback connection string when `DATABASE_URL` is absent; deployment validation must ensure production fails before relying on that fallback.
- Several privileged or high-volume routes named by the review still need rate-limit parity; the authorization and audit gates do not substitute for throttling.
- Runner defense-in-depth beyond Docker/runc (gVisor/Kata/Firecracker), Docker-socket isolation, per-tenant queue fairness, and cross-job UID separation remain infrastructure work, not application-level fixes.
- PostgreSQL capability dropping, all-stage Docker builds in CI, and additional hardening for the compose systemd unit remain operational backlog unless a deployment-specific control supplies equivalent protection.
- Live-provider AI quality evaluation, full OAuth/privileged real-cookie browser coverage, physical-device/manual assistive-technology checks, production deployment evidence, and human review of the course corpus remain outside this local remediation claim.

## Release interpretation

The High-severity remediation score is therefore **18 fixed locally / 2 partial release gates / 0 unaddressed without a recorded plan**. The application must not be described as production-complete until the two partial gates, the final repository test gate, operational configuration, and human curriculum review are reported separately with real evidence.
